# Pixel Clean

Pixel Clean 是一款面向像素图素材的本地桌面清理工具，用于把统一背景上的 AI 生成图或扁平图案转换为可复用的透明 PNG。应用优先保证输出效果：自动识别背景、恢复被背景污染的半透明边缘，并把大量近似 RGB 规整为有限且稳定的像素调色板。

项目使用 Tauri、React 和 TypeScript，当前支持 Windows 与 macOS。图片处理全部在本地 Worker 中完成，不上传原图；工程打开、保存位置选择和文件写入使用系统原生能力。

## 核心能力

- 自动估计四角背景色，也允许用户手工覆盖；
- 自然边缘输出连续 Alpha，硬像素边缘只输出 `0/255`；
- 清除圆环、边框等封闭区域中的背景；
- 使用保护画笔阻止黑色主体或重要细节被误删；
- 把实体 RGB 规整到 `8..20` 种，不产生抖动噪点；
- 按使用次数或由深到浅查看最终实体色；
- 在棋盘格、黑色、白色和自定义底色上检查透明结果；
- 参数停止变化 `500 ms` 后自动执行完整处理；
- 保存包含原图、蒙版和编辑参数的可恢复工程。

## 处理管线

```text
原始 PNG
→ 背景色和背景波动范围估计
→ 保护蒙版
→ 双阈值背景区域识别
→ 前景代表色提取
→ 边缘 Alpha 反算和 RGB 去污染
→ 相似颜色聚类与实际像素替换
→ RGB 色卡统计
→ 预览和透明 PNG 导出
```

预览、颜色统计和导出始终使用同一份处理结果。切换预览底色只改变画布显示，不会重新处理图片，也不会把底色写进 PNG。

## 核心算法一：智能背景清除

### 要解决的问题

简单的背景移除通常只计算像素 `C` 与背景色 `B` 的 RGB 距离，小于阈值就删除。这个方法会产生两类明显缺陷：

1. 抗锯齿、模糊和辉光边缘已经混入背景颜色，它们与纯背景距离较远，因此会残留为黑边或深色边；
2. 主体本身可能包含与背景接近的黑色，单纯提高容差会把主体一起删除。

Pixel Clean 将背景区域识别和边缘恢复分开处理。高可信背景负责确定透明区域，混色边缘则通过前景反算恢复 RGB 与 Alpha。

### 1. 背景颜色模型

导入图片时，从四个角的采样区域分别收集 RGB，并使用各通道中位数估计背景色。中位数比均值更不容易受角落中的前景线条或噪点影响，用户也可以通过颜色控件覆盖该结果。

处理阶段将背景色和角落样本转换为 OKLab。算法排除明显偏离背景的样本，再根据剩余样本的 90% 分位距离估计背景波动，并生成两个阈值：

- 严格阈值：确定高可信背景种子；
- 宽松阈值：允许与背景种子连通的轻微噪点和渐变继续扩展。

界面中的“背景识别范围”只影响基础范围，样本波动仍会自动参与最终阈值计算，因此正常使用不要求反复调参。

### 2. 双阈值区域生长

关闭“移除封闭背景”时，算法只从图像四边的严格背景种子开始，使用四邻域向宽松候选扩展。这样即使主体内部出现相近颜色，只要它不与外部背景连通，也不会被直接删除。

开启“移除封闭背景”时，内部严格候选也可以成为种子，用于清理圆环、边框和镂空结构中的背景。保护蒙版优先级最高：被保护的像素不能成为背景种子，也不会被区域生长覆盖。

### 3. 自动提取前景代表色

背景蒙版确定后，算法从明显远离背景的实体像素中提取最多 `48` 个全局前景候选。候选按背景距离和出现次数加权排序，优先保留稳定、纯净且常用的颜色。

全局候选不足以覆盖复杂多色图案，因此每个边缘像素还会在六像素邻域中向主体内部寻找最多 `32` 个局部候选。局部候选必须比当前像素更远离背景，避免把已经污染的暗边再次当作纯前景。

### 4. 线性 RGB 边缘反算

对背景边界内外三像素宽的区域，设输入图上观察到的颜色为 `C`、背景色为 `B`、候选纯前景色为 `F`。正常透明合成满足：

```text
C ≈ αF + (1 - α)B
```

算法先把 sRGB 转换为线性 RGB，再用投影求出最合理的覆盖率 `α`。不能直接在 sRGB 数值上计算，因为 sRGB 是非线性的，直接反算会让中间亮度产生系统性偏差。

一个候选只有同时满足以下条件才会被接受：

- `F` 比 `C` 明显更远离背景，能够代表更纯的前景；
- 求得的 Alpha 位于有效范围；
- 使用 `F`、`B` 和 Alpha 重新合成出的颜色与 `C` 误差足够小。

命中后，输出 RGB 使用未受背景污染的 `F`，而不是继续保留暗边颜色 `C`。无法可信重建的像素保持原状，避免为了追求透明而无依据破坏细节。

### 5. 自然边缘与硬像素边缘

- 自然边缘：输出反算得到的连续 Alpha，适合抗锯齿、柔和像素边缘和轻度辉光；
- 硬像素边缘：以 50% 覆盖率为分界，最终 Alpha 只保留 `0` 或 `255`。

如果输入 PNG 已经包含半透明像素，自然模式会保留其原始 RGB/Alpha，不做二次反算。所有完全透明像素最终规范化为 `#00000000`，防止透明区域携带不可见脏 RGB。

### 算法边界

当黑色主体和黑色背景在颜色与空间上都无法区分时，不存在可靠的纯颜色算法可以自动判断语义。此时应使用保护画笔明确标记主体。当前算法也不面向毛发、烟雾、玻璃或复杂摄影背景抠图。

## 核心算法二：近似颜色替换

### 要解决的问题

AI 生成的像素图即使视觉上只有红、紫、白和黑，也可能因为模糊、压缩或插值包含数千个相似 RGB。直接保留这些颜色会导致调色板庞大、难以编辑，也不符合像素图对稳定色块的要求。

颜色规整只处理 Alpha 大于 `0` 的像素。完全透明像素不参与调色板；同一 RGB 的不同 Alpha 共享一个实体颜色，Alpha 不占用 RGB 色数。

### 1. 精确颜色直方图

算法先遍历所有非透明像素，以 24 位 RGB 为键统计出现次数。像素次数会作为后续聚类权重，使大面积主体色比少量噪点颜色更有影响力。

当唯一 RGB 数量没有超过用户设置的上限时，不进行替换。当前颜色上限范围为 `8..20`，默认 `16`。

### 2. OKLab 感知距离

sRGB 数值的欧氏距离不等于人眼感知色差，因此每个唯一 RGB 都会转换为 OKLab，聚类和最近颜色选择使用：

```text
d² = (L₁ - L₂)² + (a₁ - a₂)² + (b₁ - b₂)²
```

这比直接比较 `R/G/B` 更能保持明度和色相上的视觉关系。

### 3. 色系硬约束

仅使用 OKLab 距离仍可能让深红映射到距离相近的紫色。为避免跨色系替换，算法先根据 OKLab 色度和色相把颜色分为七类：

- 中性色；
- 红色；
- 暖色；
- 绿色；
- 青色；
- 蓝色；
- 紫色。

每个实际存在的色系至少获得一个聚类中心。聚类分配、代表色选择和最终替换都只能发生在同一色系内部，因此红色像素不会被替换为紫色代表色。

### 4. 确定性加权聚类

聚类过程不使用随机初始化，确保同一输入和参数重复处理得到完全相同的结果：

1. 每个色系先选择出现次数最多的颜色作为基础中心；
2. 剩余中心使用确定性的远点策略选择，评分同时考虑与现有中心的距离和像素次数；
3. 在 OKLab 中执行 `12` 轮带像素次数权重的中心更新；
4. 每个颜色只分配给同色系的最近中心；
5. 每个聚类从原图实际存在的 RGB 中选择最接近中心的颜色作为代表色。

聚类中心只用于计算，最终输出绝不使用取整后的虚构中心色。代表色必须来自图片已有 RGB，避免算法凭空制造新的偶然颜色。

### 5. 实际替换与统计

确定代表色后，算法建立“原始 RGB → 代表 RGB”映射，并逐像素修改输出图片。Alpha 保持不变，不使用抖动，也不会通过交错噪点模拟额外色阶。

右侧色卡按最终实体 RGB 合并统计：同一 RGB 的不透明和半透明像素显示为一张色卡并累加次数，完全透明背景不显示。列表可以按像素次数或 OKLab 明度由深到浅排序。

## 预览与编辑

- `100%` 表示一个原始像素对应一个 CSS 像素；
- `- / +` 使用 `1x..16x` 整数倍率，滚轮允许连续浏览缩放；
- Canvas 保持原始像素缓冲区并使用 Point/Nearest-Neighbor 显示；
- 参数或保护蒙版变化后，停止操作 `500 ms` 自动重新处理；
- “强制高质量渲染”可以立即执行完整处理；
- `Command/Ctrl + S` 保存工程；
- `Command/Ctrl + Z` 撤销，`Command/Ctrl + Shift + Z` 或 `Ctrl + Y` 重做。

## 工程文件

当前工程格式版本为 `3`。工程保存原始 PNG、背景设置、边缘模式、保护蒙版、颜色规整设置、缩放位置和预览背景。处理结果和颜色统计不会重复写入工程，因为它们可以由输入与参数确定性重建。

旧版 v1、v2 工程可以继续打开；旧辉光参数只用于兼容读取，不再参与当前处理流程。

## 开发运行

环境要求：

- Node.js 与 npm；
- Rust 工具链（`rustc`、`cargo`）；
- macOS 需要 Xcode Command Line Tools；
- Windows 需要 Tauri 对应的 WebView2 和 C++ 构建环境。

安装依赖并启动桌面开发模式：

```sh
npm install
npm run tauri:dev
```

只检查 TypeScript 和前端生产构建：

```sh
npm run build
```

## 打包

必须在对应操作系统上构建对应的安装包：

```sh
npm run tauri:build
```

Windows 会生成 Windows 安装包；macOS 会在 `src-tauri/target/release/bundle/` 中生成 `.app` 和相应分发包。Windows 不能直接产出可发布的 macOS 应用，需要使用 Mac 或 macOS CI 运行器。

## macOS 发布

开发和本地使用可以直接构建。若要将应用分发给其他 Mac 用户，应使用 Apple Developer 证书进行代码签名并完成公证，否则 Gatekeeper 可能阻止用户打开应用。

更完整的处理约束、工程格式和验收标准见 [智能背景移除与像素颜色规整设计](docs/glow-extraction-design.md)。
