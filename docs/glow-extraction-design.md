# 辉光半透明像素提取设计

## 1. 背景与目标

AI 生成的像素图通常以统一背景色输出。普通背景剔除可以将背景变为透明，但对发光粒子、魔法特效、霓虹边缘等素材会产生错误：辉光像素已经与背景混合，直接删除接近背景的颜色会让辉光缺失、变脏或形成硬边。

本功能的目标是将“已经叠加在统一背景上的辉光”转换为可复用的 RGBA 像素：

- 纯背景变为完全透明；
- 主体核心保持不透明；
- 辉光边缘输出为半透明像素；
- 导出到深色、浅色或游戏场景背景时，不出现原背景颜色残留的暗边；
- 用户能通过局部画笔限制算法范围，避免误处理主体中的深色像素。

本功能命名为“辉光提取”，作为背景移除前的增强处理步骤，而不是替换现有背景移除功能。

## 2. 问题定义与边界

### 2.1 已知条件

- 输入是 PNG，通常没有透明通道或透明通道不可信；
- 背景色基本统一，可由四角取样或用户吸管指定；
- 用户可以识别出辉光核心和辉光覆盖区域；
- 输出为带 Alpha 通道的 PNG。

### 2.2 数学模型

一个在背景 `B` 上以正常透明混合绘制的前景像素，观察到的颜色为：

```text
C = α × F + (1 - α) × B
```

其中：

- `C`：输入图上观察到的颜色；
- `B`：已知背景颜色；
- `F`：辉光在透明图中的原始颜色；
- `α`：辉光透明度，范围为 `0..1`。

单个 `C` 无法同时唯一求出 `F` 与 `α`。因此，软件需要用户提供或自动推断 `F`，例如从辉光最亮的核心像素吸取颜色。这个限制必须明确展示在产品说明与界面提示中，不能承诺“无约束自动还原真实透明度”。

### 2.3 非目标

首版不解决以下问题：

- 从复杂摄影背景中抠出半透明毛发、烟雾或玻璃；
- 还原未知多层特效的全部独立图层；
- 自动判断任意黑色主体与黑色背景的语义差异；
- 自动修复原图中已经被压缩、模糊或手工涂抹损坏的边缘。

## 3. 用户故事

### 3.1 发光粒子制作者

作为游戏特效制作者，我导入一张黑色背景上的紫色能量球。我希望涂抹能量球外围的辉光，吸取中心的紫色，然后导出透明 PNG；放到白色或深色场景中时，辉光仍然自然。

### 3.2 黑色主体保护

作为图标制作者，我的图案中有黑色轮廓或阴阳鱼。我希望这些黑色区域不被背景剔除或辉光算法误认为透明背景，因此需要继续使用保护画笔。

### 3.3 可复现编辑

作为素材制作人员，我希望保存工程后能恢复原图、辉光涂抹区域、颜色、参数、缩放与画笔状态，以便持续修改。

## 4. 交互设计

### 4.1 工具栏与面板

在“背景移除”面板下新增“辉光提取”面板：

```text
辉光提取
[开关] 启用辉光提取
背景色： [色块] #1C1523  [吸取背景]
辉光色： [色块] #E85CFF  [从画布吸取]
混合模式： (正常透明混合) (加色辉光，实验性)
透明度强度： ─────●─────
最小透明度： ──●────────

工具： [拖动画布] [辉光画笔] [清除辉光]
画笔直径： ───●──────
已标记辉光： 1,240 px  [清除]
```

“辉光画笔”与现有“保护画笔”是两套独立的蒙版：

- 保护蒙版：该区域不参与背景剔除和辉光提取；
- 辉光蒙版：该区域允许计算半透明 Alpha；
- 两者重叠时，保护蒙版优先，保护像素保持原始不透明状态；
- 两种蒙版均使用画笔和清除画笔，均按原始像素坐标保存。

### 4.2 推荐操作流程

1. 导入图片，确认背景色取样正确。
2. 先使用保护画笔涂抹不应透明化的黑色主体、轮廓或文字。
3. 开启“辉光提取”，使用辉光画笔覆盖主体外侧的光晕，不必精确到每个像素。
4. 用吸管点击辉光最亮且最纯净的核心像素，作为辉光色。
5. 在黑色、白色、棋盘格和自定义背景之间切换预览。
6. 微调透明度强度、最小透明度或用清除辉光画笔修正局部。
7. 导出透明 PNG 或保存工程。

### 4.3 预览要求

预览区域必须提供以下背景选项：

- 透明棋盘格；
- 纯黑；
- 纯白；
- 自定义纯色；
- 原始背景。

必须提供“原图 / Alpha 预览 / 最终合成”三种视图：

- 原图：查看输入像素；
- Alpha 预览：黑色代表透明，白色代表不透明，灰色代表半透明；
- 最终合成：在当前选择的预览背景上显示结果。

仅在画笔工具启用时显示蒙版覆盖色，切回拖动画布后隐藏覆盖色，避免影响对最终效果的判断。

## 5. 算法设计

### 5.1 色彩空间

所有透明度反算必须在线性 RGB 空间完成，不能直接在 sRGB 数值上计算。流程如下：

```text
sRGB 输入像素
→ sRGB 转线性 RGB
→ 透明度与去背景计算
→ 线性 RGB 转 sRGB
→ 输出 PNG RGBA
```

原因是 PNG 中常见的 sRGB 数值是非线性的；在其中直接做 Alpha 混合会导致边缘偏暗或偏亮。

### 5.2 正常透明混合模式（首版正式支持）

用户选择辉光原始颜色 `F`，背景色为 `B`，输入像素为 `C`。在辉光蒙版内用最小二乘投影计算透明度：

```text
αraw = dot(C - B, F - B) / dot(F - B, F - B)
α = clamp(αraw × strength, 0, 1)
```

其中 `strength` 来自界面中的“透明度强度”滑条，默认值为 `1.0`。

输出规则：

```text
输出 RGB = F
输出 Alpha = round(255 × α)
```

附加规则：

- 若像素在保护蒙版内：保持原始 RGB 与 Alpha `255`；
- 若 `α` 小于“最小透明度”阈值：输出 Alpha `0`；
- 若 `α` 大于“核心不透明阈值”：可输出 Alpha `255`，但首版默认关闭此项，避免硬边；
- 若投影为负值，视为背景或与辉光方向不一致，输出透明；
- 用户可以对辉光蒙版使用清除画笔，剔除误选区域。

该模式适合单色或颜色变化较小的火焰、光球、魔法阵外圈、霓虹轮廓。

### 5.3 加色辉光模式（实验性）

部分 AI 图或游戏截图使用近似加色混合：

```text
C = clamp(B + α × G, 0, 1)
```

其中 `G` 为发光增量颜色。该模式不能与普通 Alpha 完全等价，但可通过用户选择的辉光色估算一个适合普通 RGBA 输出的近似结果。

首版仅提供实验性开关，并在界面显示提示：

> 加色辉光的导出效果依赖目标引擎的混合方式；请在目标引擎中验证。

### 5.4 多色辉光（第二阶段）

当辉光核心包含蓝紫、粉紫、白色高光等多种颜色时，单一 `F` 会丢失渐变。第二阶段方案：

1. 用户在核心区域用吸管添加 2–8 个辉光色；
2. 对辉光蒙版中的每个像素，选择与 `C - B` 方向最接近的辉光色；
3. 使用该颜色作为 `F` 计算 Alpha；
4. 可选地使用空间平滑约束，避免相邻像素在色卡间频繁跳变。

该方案应与未来的色卡功能共享颜色选择控件。

## 6. 处理管线

完整处理顺序如下：

```text
原始 PNG
→ 背景色估计/用户吸取
→ 保护蒙版应用（优先级最高）
→ 辉光蒙版 Alpha 提取
→ 普通背景剔除（跳过保护蒙版和辉光蒙版）
→ 可选色卡规整
→ RGBA PNG 导出
```

现有“移除封闭背景”功能在启用辉光提取后必须遵循以下规则：

- 辉光蒙版中的像素不能被直接置为 Alpha `0`；
- 算法先为这些像素生成半透明 Alpha；
- 保护蒙版覆盖辉光蒙版时，始终按保护蒙版处理；
- 其它相似背景色仍可按现有全图或边缘连通策略移除。

## 7. 数据模型与工程文件

现有工程文件应从版本 `1` 升级到版本 `2`，新增以下字段：

```json
{
  "kind": "pixel-clean-project",
  "version": 2,
  "source": {
    "fileName": "particle.png",
    "pngDataUrl": "data:image/png;base64,..."
  },
  "editor": {
    "tolerance": 20,
    "includeEnclosedAreas": true,
    "imageView": { "scale": 4, "offsetX": 12, "offsetY": -18 },
    "brushSize": 12,
    "interactionMode": "paint",
    "protectedMask": "base64...",
    "glowExtraction": {
      "enabled": true,
      "backgroundColor": "#1C1523",
      "foregroundColors": ["#E85CFF"],
      "blendMode": "normal",
      "strength": 1,
      "minimumAlpha": 0.02,
      "glowMask": "base64...",
      "brushSize": 10
    }
  }
}
```

迁移规则：

- 版本 `1` 工程打开时，自动补充 `glowExtraction.enabled = false`；
- 版本 `2` 工程必须校验蒙版长度是否等于 `width × height`；
- 任何字段校验失败时，拒绝加载并保留当前工程，不覆盖现有编辑状态。

## 8. 实施拆分

### 阶段 A：最小可用版本

1. 增加辉光蒙版状态、画笔、清除画笔与工程持久化。
2. 增加辉光色吸管、背景色手动吸管。
3. 实现线性 RGB 转换与单色正常透明混合计算。
4. 修改背景剔除管线，使其跳过辉光蒙版。
5. 增加 Alpha 预览和黑白背景预览。

### 阶段 B：质量与可控性

1. 增加透明度强度、最小透明度、核心不透明阈值。
2. 增加加色辉光实验模式。
3. 增加撤销/重做，记录辉光画笔笔划和参数变化。
4. 为大图使用分块计算或 Web Worker，避免画笔绘制卡顿。

### 阶段 C：多色与自动建议

1. 从核心区域提取多色辉光色卡。
2. 自动建议候选辉光区域与辉光色。
3. 以色彩连续性约束优化多色辉光边缘。
4. 批处理多个使用同一背景与辉光规则的素材。

## 9. 验收标准

### 9.1 功能验收

- 用户能用画笔涂抹任意形状的辉光区域，并用清除画笔局部修正。
- 用户能从画布选择背景色与辉光核心色。
- 用户能在黑色、白色、棋盘格和自定义背景上预览。
- 导出的 PNG 具有真实的 `0..255` Alpha 值，不只是二值透明。
- 关闭并重新打开工程后，辉光蒙版、颜色、参数、缩放、平移与画笔大小完整恢复。

### 9.2 视觉验收

使用黑底紫色光球、白底蓝色火焰、带黑色主体的霓虹图标三类样本：

- 在白色背景上，不出现黑色或原背景颜色边缘；
- 在黑色背景上，辉光轮廓连续，不被硬切；
- 保护区域内的黑色主体仍保持不透明；
- Alpha 预览中，辉光从核心到外缘呈连续或符合像素风设计的阶梯式衰减；
- 放大至 `1600%` 时，输出像素与预览像素一致。

### 9.3 性能验收

- `256 × 256` 图像在拖动画笔、调整参数时预览响应小于 `50 ms`；
- `2048 × 2048` 图像首次计算小于 `1 s`，参数调整小于 `300 ms`；
- 处理过程不改变原始图像数据，所有操作可通过工程文件复现。

## 10. 风险与决策

| 风险 | 影响 | 处理方式 |
| --- | --- | --- |
| 辉光原始颜色未知 | Alpha 无法唯一反推 | 要求用户从核心吸取颜色，提供预览校正 |
| 黑色主体与黑色背景接近 | 主体被透明化 | 保护画笔优先级最高 |
| AI 图不是正常 Alpha 混合 | 结果与原视觉存在偏差 | 提供强度参数与实验性加色模式 |
| 大面积蒙版重复计算 | 画笔操作卡顿 | 阶段 B 使用 Web Worker、脏区域更新或 Rust 后端计算 |
| 多色辉光 | 单色输出丢失渐变 | 阶段 C 支持多色辉光色卡 |

## 11. 评审结论

建议按阶段 A 实施。它以“用户圈定区域 + 吸取辉光核心色”解决了反算透明度的不可确定性，能稳定覆盖游戏粒子和 AI 像素图最常见的辉光场景。保护画笔、辉光画笔、黑白背景预览与工程持久化应被视为同一功能的必要组成，而不是可选增强。
